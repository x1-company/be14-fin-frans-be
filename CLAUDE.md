# CLAUDE.md — FRANS Refactoring

LLM 코딩 실수를 줄이기 위한 행동 지침. 프로젝트 규약과 함께 적용한다.
**Tradeoff:** 속도보다 신중함에 가중치를 둔다. 사소한 작업은 판단에 맡긴다.

## 0. 프로젝트 컨텍스트

- 프랜차이즈 주문관리 ERP. Spring Boot 3.2.5 / Java 17 / MariaDB / Redis.
- 아키텍처: **CQRS** — command는 JPA, query는 MyBatis(`Mapper.xml`).
- 목표: 6인 팀 프로젝트를 **단독 리팩토링**해 도메인 설계·동시성·회복탄력성·AI 활용을 증명.
- 주석·커밋·문서는 한국어. 기존 패키지 컨벤션(`domain/command/query`) 유지.

## 1. Think Before Coding

가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.

- 가정은 명시한다. 불확실하면 묻는다.
- 해석이 여러 갈래면 말없이 하나 고르지 말고 제시한다.
- 더 단순한 길이 있으면 말한다. 필요하면 반대한다.

## 2. Simplicity First

문제를 푸는 최소 코드. 투기적 추상화 금지.

- 요청 안 한 기능·유연성·설정 추가 금지.
- 패턴은 **악취를 제거할 때만** 도입한다(패턴을 위한 패턴 금지).
- 200줄이 50줄로 가능하면 다시 쓴다.

## 3. Surgical Changes

건드려야 할 것만 건드린다. 내가 만든 것만 치운다.

- 인접 코드·포맷을 "개선"하지 않는다.
- 깨지지 않은 걸 리팩토링하지 않는다.
- 기존 스타일에 맞춘다. 발견한 무관한 dead code는 **삭제 말고 보고**한다.

## 4. Goal-Driven Execution (리팩토링 특화)

성공 기준을 정하고 검증될 때까지 반복한다.

- **"Refactor X" → "리팩토링 전 특성화 테스트로 현재 동작을 고정하고, 전·후 모두 통과시킨다."**
- "버그 수정" → 재현 테스트부터 작성 후 통과.
- 다단계 작업은 `단계 → 검증` 형식으로 계획을 먼저 적는다.

## 5. Refactoring 불변식 (이 프로젝트의 핵심)

- **외부 동작 보존**: 리팩토링은 API 응답·DB 상태를 바꾸지 않는다. 바뀐다면 그건 리팩토링이 아니라 기능 변경이며 별도로 보고한다.
- **도메인 우선**: 상태전이·검증은 서비스가 아니라 애그리거트(`Approval`, `Order`)에 둔다.
- **트랜잭션 경계**: 외부 호출(LLM/메일/알림)은 트랜잭션 안에서 하지 않는다. 도메인 이벤트 `AFTER_COMMIT`로 뺀다.
- **Spring `@Transactional`만** 사용(`jakarta.transaction` 금지).
- **매직스트링·매직넘버 금지**: enum/상수로.

## 6. 작업 방식 (사용자 선호)

- 코드는 **직접 수정하지 말고 블록으로 제안**한다(사용자가 직접 반영하며 학습).
- git 명령을 자동 실행하지 않는다.
- 큰 변경 전 단계별 계획을 먼저 합의한다.
